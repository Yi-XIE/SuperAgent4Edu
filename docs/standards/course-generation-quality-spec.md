# 课程生成质量执行规范（内部版）

日期：2026-03-14

适用对象：

- `Lead Agent`
- `Blueprint Agent`
- `Package Agent`
- reviewer / critic

关联目录：

- `docs/plans/`
- `docs/prompts/`
- `docs/cases/`

相关 prompt：

- `docs/prompts/Reviewer-critic-prompts.md`

## 1. 文档目的

这份文档用于定义课程生成任务的内部质量底线和 review 机制。

本规范不是课程模板，也不是标准答案。它不规定一份课程必须采用哪种情境、哪种文风、哪种章节顺序，也不规定必须出现某几个固定活动。

它只负责回答两个问题：

1. 哪些质量底线必须守住
2. reviewer / critic 应该如何判断是否需要返工

## 2. 总原则

### 2.1 管质量，不管写法

本规范控制的是课程质量，不是课程外形。

允许变化的内容包括：

- 课程情境
- 文风
- 章节顺序
- 活动类型
- 课堂组织方式
- 学具形态

前提是这些变化不能破坏本规范中的不可妥协项。

### 2.2 先自由生成，再按 rubric 评审

推荐流程如下：

1. 生成 agent 先根据任务自由产出
2. reviewer / critic 按本规范检查
3. 不达标则局部返工
4. 返工依据是原则，不是样板句式

### 2.3 案例用于校准，不用于复写

案例库的用途是帮助团队识别“高质量课程通常具备哪些特征”，不是让 agent 复写某一个案例的外形。

禁止以下做法：

- 要求输出必须像某个案例一样分成相同章节
- 要求必须采用某个案例的活动数量或活动顺序
- 因为某个案例成功，就把其情境、道具或表达方式当成通用模板

## 3. 不可妥协项

以下 4 项是硬门槛。任何一项明显不达标，都不应直接通过。

### 3.1 目标、证据、活动要一致

含义：

- 学习目标必须能在评价证据中被看见
- 评价证据必须能在活动过程中被支撑
- 活动不能只是热闹，必须真正服务目标与证据

通过信号：

- 课程蓝图中的核心理解能在证据设计中找到对应证据
- 证据设计中的表现性任务能在活动流程中找到准备路径
- 完整课包整理后，不会出现“目标写一套，课堂上一套”的断裂

失败信号：

- 目标写得很完整，但评价只剩泛泛展示
- 活动流程丰富，但看不出与目标的关系
- 学生最后的产出无法证明核心理解是否达成

默认返工路由：

- 方向和目标本身有问题：回退 `Blueprint Agent`
- 证据设计、活动承接或课包实现有问题：回退 `Package Agent`

### 3.2 驱动问题要儿童可理解

含义：

- 问题表达要符合对应学段的理解能力
- 问题既要有探究价值，也要让孩子听得懂、愿意答
- 复杂的大概念可以保留在教师设计层，但面向学生的驱动问题必须可进入

通过信号：

- 小学生能在首次接触时大致明白问题在问什么
- 问题能引出观察、比较、制作、解释或展示
- 问题与真实体验之间有桥梁，不是纯抽象口号

失败信号：

- 问题词汇过学术，像论文标题
- 问题过大，学生无法形成具体行动
- 问题只有“好听”，没有实际课堂驱动力

默认返工路由：

- 问题层级不合适：回退 `Blueprint Agent`
- 活动不能承接问题：同步检查 `Package Agent`

### 3.3 学具必须服务教学目标

说明：

- 如果本轮任务包含实体学具或体验材料，这一项必须检查
- 如果本轮明确不需要学具，可标记为 `N/A`

含义：

- 学具不是装饰，不是为了显得有趣而附加
- 每个学具都必须对应某个教学目标和课堂环节
- 学具价值应体现在帮助学生观察、比较、体验、建模、表达或验证

通过信号：

- 能明确说明学具服务哪个目标、哪个问题、哪个环节
- 学具使用方式和课堂任务是连在一起的
- 材料、发放、回收、安全与替代方案可说明白

失败信号：

- 学具看起来有创意，但与学习目标关系弱
- 学具附录与正文分离，像额外清单
- 材料复杂到脱离学校现实

默认返工路由：

- 学具与目标脱节：回退 `Package Agent`
- 活动环节定义不够：仍优先回退 `Package Agent`

### 3.4 课时流程要真实可执行

含义：

- 课时安排要符合小学课堂的时间、组织和管理现实
- 活动节奏要可落地，而不是理想化拼贴
- 教师需要能看出每一课大致怎么推进

通过信号：

- 课时结构清楚，导入、探究、展示、总结关系明确
- 分组、材料、空间和时间安排有现实感
- 课堂里需要的关键转场是可操作的

失败信号：

- 单课时塞入过多目标和活动
- 重要环节缺时间，次要环节过度展开
- 课堂管理成本极高，但文档没有任何处理

默认返工路由：

- 节奏、组织或课包实现问题：回退 `Package Agent`
- 只是返工方向不清或约束需重开：回退 `Lead Agent`

## 4. 明确保留的开放空间

以下内容不应被写死，否则会压制 agent 发挥。

### 4.1 不规定必须用哪种情境

允许使用：

- 故事情境
- 真实问题情境
- 科学探究情境
- 工程设计挑战
- 游戏化或展览式情境

判断标准不是“像不像某个案例”，而是“是否服务当前课程目标”。

### 4.2 不规定必须写成哪种文风

允许结果偏向：

- 教师工作文档
- 课堂脚本
- 课程设计卡
- 项目任务书
- 展示说明稿

前提是表达清楚、教师可用。

### 4.3 不规定固定章节顺序

结果包可以根据场景调整顺序，例如：

- 先写课程概览，再写 UbD 三阶段
- 先写项目任务，再回扣证据与活动
- 先展示教案，再附课程设计卡

只要目标、证据、活动关系没有被打散即可。

### 4.4 不规定必须出现固定活动

禁止因为某个案例有效，就默认每节课都必须包含：

- 工作站轮换
- 展览会
- 海报制作
- 角色扮演
- 某种固定学具

活动形式应由课程内容和学段需求决定。

## 5. 课程质量 Rubric

### 5.1 评分说明

每个维度按 `0-3` 分评估：

- `0`：缺失或明显不成立
- `1`：有相关内容，但质量较弱或存在明显断裂
- `2`：达到通过线，基本可用
- `3`：质量强，支撑关系清楚，具有较好教学适配性

### 5.2 维度说明

| 维度 | 是否硬门槛 | reviewer 关注点 | 常见低分信号 |
|---|---|---|---|
| 目标-证据-活动一致性 | 是 | 目标是否能被证明，活动是否为证据做准备 | 目标、评价、活动彼此脱节 |
| 驱动问题儿童可理解性 | 是 | 问题是否听得懂、答得动、能驱动探究 | 问题空泛、抽象、过学术 |
| 学具服务教学目标 | 条件性硬门槛 | 学具是否真正服务目标与课堂环节 | 学具有趣但无教学功能 |
| 课时流程现实可执行 | 是 | 时间、组织、转场、材料管理是否可信 | 一课时装太多内容，管理负担过高 |
| 表现性任务与证据质量 | 否 | 最终产出是否真的能看见学生理解 | 只剩作业化陈列，不能证明理解 |
| 体验与探究密度 | 否 | 学生是否真正经历观察、比较、制作、解释或测试 | 活动偏讲授，体验过薄 |
| 差异化与支架支持 | 否 | 是否考虑低年级表达支架、记录单或句式支持 | 所有学生被默认能自然完成高要求表达 |
| 成稿清晰度 | 否 | 教师是否容易快速读懂和修改 | 信息堆砌、结构混乱 |
| 灵活性与非模板化 | 否 | 结果是否贴合任务，而不是套同一外形 | 明显照搬某一案例结构 |

说明：

- `学具服务教学目标` 仅在本轮任务包含学具时作为硬门槛；否则标记 `N/A`
- reviewer 可以补充简短说明，但不建议扩张为过长评论

### 5.3 通过规则

建议采用以下判断：

- `通过`：所有硬门槛维度均达到 `2` 分及以上；其余维度整体稳定
- `有条件通过`：硬门槛达标，但存在 1 到 3 个明显可优化项
- `不通过`：任一硬门槛未达标，或整体质量不稳定

## 6. Reviewer / Critic 检查顺序

建议固定按以下顺序检查：

1. 先读任务约束，不先读案例
2. 快速判断 4 个硬门槛是否成立
3. 再按 rubric 给出简短评分或判断
4. 确定问题属于哪个运行层的职责范围
5. 给出局部返工建议

### 6.1 反馈写法要求

反馈应引用原则，不引用样板外形。

推荐写法：

- 当前驱动问题对二年级学生仍偏抽象，建议回到儿童可理解表达
- 学具与目标关联不清，学生使用后无法自然生成目标所需证据
- 第二课时展示任务与前一课时探究活动衔接不足

不推荐写法：

- 请改成像案例 01 那样的三个问题
- 请像某个示范一样做三个工作站
- 请套用某份优秀教案的结构

### 6.2 Reviewer 不能代替上游修问题

当问题出在上游时，reviewer 不应在下游偷偷修平。例如：

- 课程蓝图的目标有问题，不应要求完整课包仅靠整理掩盖
- 证据设计有问题，不应要求活动流程靠增加活动补救

## 7. 返工路由

| 问题类型 | 优先回退对象 | 说明 |
|---|---|---|
| 大概念、驱动问题、目标层级不合适 | `Blueprint Agent` | 先修课程蓝图层 |
| 证据、表现性任务、活动流程、学具附录或整理不对应目标 | `Package Agent` | 先修完整课包层 |
| 多模块冲突、返工方向不清或约束需重开 | `Lead Agent` | 由主控决定局部重跑顺序 |

## 8. 案例库使用规则

### 8.1 正确使用方式

使用案例库时，应先抽取共性原则，再开始生成或 review。建议顺序如下：

1. 选择 `2-3` 个风格不同的优质案例
2. 提炼这些案例共同成立的质量信号
3. 用这些质量信号指导生成或 review
4. 不直接照抄案例章节、标题、道具或活动编排

### 8.2 当前案例数量不足时的处理

理想状态下，案例库应有 `3-5` 个风格不同的高质量案例。

如果当前少于 `3` 个案例，reviewer 应明确知道：

- 当前校准基础偏薄
- 可用案例只适合辅助判断，不适合替代 rubric
- 生成时更要依赖本规范，而不是依赖单个案例

### 8.3 案例入库标准

案例进入文件夹前，至少要满足以下条件：

- 能看出清楚的目标、证据、活动关系
- 具有课堂可执行性，不只是概念展示
- 有足够细节支持团队分析“为什么有效”
- 与现有案例相比，提供新的主题、年级、活动或表达类型

### 8.4 明确禁止事项

禁止把案例库用于以下用途：

- 规定固定章节顺序
- 规定固定活动数量
- 规定固定驱动问题句式
- 规定固定学具形态
- 用一个案例覆盖所有主题和学段

## 9. 反模式

以下现象属于反模式，应被明确避免：

- 用“像不像案例”代替“是否达标”
- 用文风整齐代替教学逻辑清楚
- 用新奇学具代替真实学习证据
- 用复杂活动堆砌代替可执行课时结构
- 用 Presentation 润色掩盖上游设计缺陷
- 因为担心发散，就把 agent 的输出外形全部写死

## 10. 当前阶段建议

在当前项目阶段，建议先做到以下三件事：

1. 让 reviewer / critic 明确按照本规范工作
2. 持续补足 `3-5` 个风格不同的案例
3. 后续再视效果决定是否把本规范中的部分原则映射进 agent prompt 或 skill
