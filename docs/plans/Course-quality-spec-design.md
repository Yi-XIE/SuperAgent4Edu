# 课程质量规范与案例库设计说明

日期：2026-03-14

## 1. 文档目的

这份设计说明用于确认一项内部规范设计决策：

- 不把单个优质课程案例直接改写成生成模板
- 不用固定章节和固定活动压制 agent 发挥
- 通过少数不可妥协项与 reviewer rubric 控制课程质量
- 单独建立案例文件夹，用于校准和抽取共性原则，而不是复写案例外形

这份文档是设计决议，不替代后续实际执行规范。

## 2. 设计判断

当前项目已经具备较清楚的 UbD/PBL 骨架、多代理分工和学具附录定位，但仍然缺少一层“如何判断结果够不够好”的内部执行规范。

如果直接把高质量案例写成 prompt 模板，容易出现以下问题：

- 输出越来越像某一份案例的复写
- agent 为了“长得像”而不是“质量达标”去生成
- reviewer 的反馈会滑向句式和章节顺序，而不是课程质量原则
- 不同主题、不同年级的课程会被强行套进相同外形

因此，这次不采用“标准答案模板”，而采用“质量底线 + rubric + 案例校准”的方式。

## 3. 设计决策

本次采用“单主文档 + 案例文件夹”的落法。

### 3.1 主文档

新增一份内部执行规范，放在：

- `docs/standards/course-generation-quality-spec.md`

这份文档负责定义：

- 少数不可妥协项
- 明确保留的开放空间
- reviewer/critic 的检查顺序
- 返工路由
- 案例的正确使用方式

### 3.2 案例文件夹

新增一组课程质量案例文件，放在：

- `docs/cases/`

这一层只负责：

- 归档高质量案例
- 说明为什么这个案例值得入库
- 抽取可迁移原则
- 明确哪些部分不能照抄

### 3.3 不做的事情

这次不做以下内容：

- 不改写现有 agent prompt
- 不把案例内容硬编码进子代理职责
- 不规定教案必须采用统一章节顺序
- 不提前把所有课程压成同一套课堂结构

## 4. 预期使用方式

建议后续按以下顺序使用：

1. 生成 agent 先基于任务自由产出课程草案
2. reviewer/critic 使用质量执行规范检查结果
3. 若存在问题，只按原则给出反馈，不引用案例外形
4. 需要返工时，只回退到真正出问题的 agent
5. 案例库只用于校准“为什么好”，不用来规定“必须长成什么样”

## 5. 当前交付范围

本次实际落地包括：

- 一份内部执行规范
- 一份案例库 README
- 一个首个入库案例
- 一个新增案例模板
