## 1. 文档目的

这份文档用于定义一个基于 DeerFlow 的教育课程工作台产品方案与演进路径。目标不是一开始就做成重平台，而是先借助一个真实、克制、可演示的教育场景跑通主流程，再逐步演进为真正可用的个人备课助手。文档重点包括：

- 多代理编排
- 记忆系统
- 技能系统
- 前端可视化展示

本方案面向一个约 20 人的教师小团队，应用方向聚焦在人工智能教育和科学教育，课程设计方法采用 UbD 和 PBL。第一版默认以小学场景为主，并补充与课程配套的实体学具规划。

## 2. 项目定位

### 2.1 产品名称

暂定名称：`UbD/PBL 教师备课与学具规划工作台`

### 2.2 产品定位

这是一个面向教师的课程设计助手，用于帮助教师围绕人工智能教育和科学教育主题，快速生成一套带有 UbD 结构和 PBL 项目逻辑的课程初稿，并给出与课堂配套的实体学具附录。

它的本质不是“万能问答机器人”，而是“课包工厂 + 教师素材台”的课程设计工作台。

### 2.3 第一阶段目标

第一阶段的核心不是追求业务完整，而是完成以下学习和演示目标：

- 看懂 DeerFlow 如何把一个教师任务拆成多个子代理协作完成
- 看懂 DeerFlow 如何在多轮会话中记住教师偏好与课程连续性
- 做出一个有前端界面的可演示 Demo
- 让产出内容看起来像真实教师可用的课程设计初稿

### 2.4 非目标

第一版明确不做以下事情：

- 不做面向学生的开放平台
- 不做复杂多租户体系
- 不做完整权限系统和组织架构
- 不做正式上线级安全治理
- 不做复杂工作流编辑器
- 不做精美 PPT 文件自动排版，先输出 PPT 大纲
- 不做大规模知识库平台，先以少量课程资料和白名单来源为主

## 3. 目标用户与使用场景

### 3.1 目标用户

- 人工智能教育教师
- 科学教育教师
- 教研员
- STEAM/项目式学习课程设计者

### 3.2 核心场景

教师输入一个课程主题后，系统生成：

- UbD 课程设计卡
- PBL 项目任务框架
- 教案初稿
- PPT 大纲
- 学具附录
- 参考资料摘要

适合的主题示例：

- 生成式 AI 素养课程
- 智能传感器与数据采集
- 科学探究与工程设计融合课程
- AI + 科学实验的跨学科项目单元

### 3.3 第一版输入输出定义

第一版默认面向小学 AI 教育和科学教育场景，采用“课程为主，学具为辅”的输入输出结构。

建议输入字段：

- 课程主题
- 适用年级
- 学科方向
- 课时数
- 课程目标
- 是否需要项目式学习
- 是否需要实验活动
- 是否需要实体学具
- 学具限制条件
- 输出偏好

建议输出包：

- UbD 课程设计卡
- PBL 项目任务框架
- 教案初稿
- PPT 大纲
- 学具附录
- 参考资料摘要

其中 `学具限制条件` 建议至少覆盖：

- 是否优先低成本材料
- 是否允许打印件
- 是否允许外包定制件
- 是否需要统一发放与回收
- 是否有明确安全限制

## 4. 为什么选择 DeerFlow

这个项目适合作为学习样板，原因在于它天然覆盖了你最关心的能力：

- `Sub-agents`：适合做备课任务拆解和并行执行
- `Memory`：适合记忆教师偏好和课程历史
- `Skills`：适合沉淀 UbD、PBL、AI 教育、科学教育的专业流程
- `Sandbox`：适合理解文件产出、工作区和工具执行模式
- `Frontend + Backend`：适合学习一个完整 Agent 产品如何对外呈现

对你的目标来说，DeerFlow 不是成品平台，而是一套值得拆开学习的运行底座。

## 5. 产品方案概览

### 5.1 一句话方案

做一个基于 DeerFlow 的教师备课工作台，让教师输入课程主题和基本要求后，系统通过多个专业子代理协作，输出一份融合 UbD 和 PBL 的完整课包，并把过程中形成的高价值教学片段沉淀到教师素材台，帮助老师越用越省力。

### 5.2 产品价值

对教师的价值：

- 降低课程设计初稿的准备时间
- 让课程结构更符合 UbD 和 PBL 方法
- 支持 AI 教育和科学教育这类新型课程场景
- 让实体学具方案直接服务课程目标，而不是脱离教学单独规划
- 让每次备课都能沉淀为下次可直接调用的教学素材

对你的学习价值：

- 真正理解多代理如何拆解任务
- 理解记忆系统何时读、何时写、写什么
- 理解技能系统如何承载专业方法论
- 理解一个 Agent 产品前端该如何展示过程和结果

### 5.3 Part 1：产品骨架和核心工作流

这个产品不再只是“一次性生成一份课”，而是围绕老师真实备课过程，长期管理两类核心对象：

- `课包`：一次备课的最终成果，包含课程卡、教案、PPT 大纲、学具附录、资料摘要
- `素材`：老师可持续复用的备课资产，包含目标片段、驱动问题、活动点子、学具方案、参考资料、表达模板

这样一来，主线仍然是“从一个主题快速生成完整课包”，但系统不再把每次任务都当成一次性问答，而是把过程中形成的高价值内容沉淀成下次可调用的素材。

核心工作流建议如下：

1. 老师新建任务，选择两种入口之一：
   - `快速生成课包`
   - `带着已有想法生成`
2. 系统先形成一张 `任务简报`，内容包括主题、年级、课时、学科方向、是否需要 PBL、是否需要学具、已有素材摘要
3. 第一审批点不再只是“确认任务”，而是同时确认 `生成策略`
   - 是偏“从零生成”
   - 还是“优先吸收已有素材”
4. `目标蓝图` 与 `Research` 完成后，系统形成 `课程蓝图`
   - 也就是大概念、核心问题、项目方向、研究支撑的组合卡
5. 老师锁定蓝图后，系统继续生成完整课包
   - 包括 UbD 三阶段、PBL 任务、教案、PPT 大纲、学具附录
6. 草案评审通过后，系统不只是结束，而是做一次 `自动资产提取`
   - 从本次课包中抽出可复用素材，存入教师素材台
7. 下次老师再备课时，系统优先调用素材台中的高相关资产，而不是只依赖长期记忆里的散碎偏好

产品结构建议如下：

- 左侧：任务入口、历史课包、素材台
- 中间：执行流、审批卡、当前生成阶段
- 右侧：课包结果区，按“课程蓝图 / 教案 / PPT / 学具 / 参考资料 / 本次提取素材”分栏展示

这一版最关键的变化，不是“再加几个 agent”，而是把系统从“会生成”升级为“会积累”。

### 5.4 Part 2：5 个补完模块

#### 5.4.1 教学内容资产模块

“教师素材台”是这版升级的核心。它不存整份课，而是存可复用的教学片段，重点包括 6 类：

- 目标片段
- 驱动问题
- 活动点子
- 学具方案
- 参考资料
- 表达模板

每条素材都应带最少元数据，例如适用年级、学科方向、适用课时、是否适合 PBL、来源课包、使用次数、老师是否标记为“好用”。

它的关键不是“收集越多越好”，而是“让系统知道什么时候该调什么”。因此素材需要两种进入方式：

- 老师主动保存
- 草案通过后系统自动提取候选素材，再由老师一键确认入库

#### 5.4.2 教师工作流模块

这一层要把现在的 Demo 对话流升级为真正的 `备课任务流`。最小闭环建议是：

1. 新建任务
2. 生成任务简报
3. 锁定生成策略
4. 形成课程蓝图
5. 生成完整课包
6. 草案评审
7. 提取素材
8. 归档

这里最重要的新动作是 `生成策略确认`，也就是系统先判断本轮是偏从零生成，还是偏吸收已有素材。

#### 5.4.3 资源复用模块

这一层的原则是：

- 下次生成时，优先查 `素材台`
- 其次查 `长期记忆`
- 最后才查 `外部资料`

长期记忆更适合保存偏好，素材台更适合保存教学内容资产，二者不能混用。

因此在任务简报和课程蓝图阶段前，系统应先做一次素材召回，给出“本轮可参考素材卡”，并让老师决定哪些要吸收进本轮。

#### 5.4.4 校本协作模块

这一层不建议先做重权限，而建议先做轻量机制：

- 个人素材默认私有
- 老师可以把某条素材或课包片段发布到 `校本库`
- 进入校本库后，变成团队可见的共享资产

团队层最适合共享的不是完整课包，而是高复用片段，例如目标表述、活动结构、学具附录写法和表达模板。

#### 5.4.5 课后反馈闭环模块

每次课包使用后，老师只需要补一个很轻的反馈：

- 哪些环节真实使用了
- 哪些地方改了
- 学生哪部分最有反应
- 哪个学具其实不好用
- 哪条驱动问题最能带动课堂

系统据此更新两件事：

- 课包的实战状态
- 素材的可信度

这样素材台里的内容就会逐渐分成三层：

- 生成素材
- 教师修订素材
- 课堂验证素材

### 5.5 Part 3：核心数据模型、页面结构与 Agent 分工

这一部分的核心原则是：

- `课包` 是交付物
- `素材` 是生产资料
- `记忆` 是偏好层

这三层必须分开。

建议的最小数据模型包括：

1. `TaskBrief`
   - 保存主题、年级、课时、学科方向、是否需要 PBL、是否需要学具、已有想法摘要、生成策略
   - 新增关键字段：`generation_mode`
   - 可选值：`from_scratch` / `material_first` / `mixed`
2. `CourseBlueprint`
   - 课程蓝图对象
   - 包含大概念、核心问题、驱动问题、项目方向、研究支撑摘要、推荐生成路径
3. `CoursePackage`
   - 一次备课的最终成果
   - 包含课程卡、UbD 三阶段、PBL 任务、教案、PPT 大纲、学具附录、参考资料摘要、来源蓝图 ID、来源素材 ID 列表
4. `TeachingAsset`
   - 教师素材台的核心对象
   - 建议素材类型：`goal_fragment` / `driving_question` / `activity_idea` / `learning_kit_plan` / `reference_note` / `expression_template`
5. `ResourceItem`
   - 外部或内部参考资料对象
   - 更偏证据与资料层，不等于教学素材
6. `AssetExtractionCandidate`
   - 每次课包通过后自动生成的候选素材
   - 老师确认前的临时层
7. `TeachingFeedback`
   - 记录课后使用、修改与复用意愿

页面结构建议如下：

- 左侧：`新建任务`、`历史课包`、`教师素材台`
- 中间：`任务流 + 审批流`
- 右侧：`结果 + 沉淀`

中间区域建议突出 7 个任务节点：

1. 任务简报
2. 生成策略确认
3. 蓝图生成中
4. 蓝图审批卡
5. 课包生成中
6. 草案评审卡
7. 素材提取确认

右侧结果区建议固定为：

- 课程蓝图
- 教案
- PPT
- 学具
- 参考资料
- 本次提取素材

运行层不建议继续细拆，而建议收敛为 `4+1` 模型：

1. `Lead Agent`
   - 负责任务简报、生成策略、审批和总控
2. `Blueprint Agent`
   - 合并课程蓝图与研究支撑
   - 对外输出老师可锁定的课程蓝图
3. `Package Agent`
   - 合并证据设计、活动流程、学具附录与最终整理
   - 对外直接输出完整课包
4. `Reviewer Agent`
   - 承担首轮质量闸门
5. `Critic Agent`
   - 建议作为可选第二层质量复核
   - 在高风险、低置信度或老师要求严格复核时启用

同时保留 2 个系统能力，但不把它们做成一等 agent：

- `Asset Retrieval`
  - 在生成前做素材召回与优先复用建议
- `Asset Extraction`
  - 在草案通过后做候选素材提取与入库建议

这样做的目的不是减少方法论，而是减少运行负担。概念上仍然可以区分“目标蓝图 / 证据设计 / 活动流程 / 学具 / 整理”，但运行时不必让每一层都独立成为 agent。

### 5.6 Part 4：最小可落地版本与演进切法

这版不建议直接做成重平台，而建议先把它做成真正可用的 `个人备课助手`，再逐步长出校本协作能力。

第一阶段的必做闭环建议是：

1. 保留当前 `项目 / 运行 / 工件 / 审批点` 骨架，继续用它承载课包生成流程
2. 在现有任务流中加入：
   - `任务简报`
   - `生成策略确认`
   - `课程蓝图`
   - `素材提取确认`
3. 保留现有 `Reviewer + Critic + 草案评审护栏`
4. 新增最小素材台：
   - 先支持个人素材
   - 先支持主动保存与自动提取候选
5. 在生成前加入一次素材召回
6. 在课包通过后加入一次素材提取
7. 让记忆系统只承担：
   - 教师偏好
   - 课程连续性
   - 团队风格
   - 不再承载可复用教学内容片段

第一阶段暂不做：

- 重权限体系
- 复杂工作流编辑器
- 复杂模板市场
- 完整学生端运营
- 学具库存、采购与供应链
- 精美 PPT 自动排版

从当前代码平滑演进的建议路径是：

1. 复用现有聊天页与审批卡，把“任务确认点”扩成“任务简报 + 生成策略确认”
2. 复用现有 `Stage 1 + Research` 结果，把它包装为老师可读的 `课程蓝图`
3. 在运行编排上，把生成层逐步收束为 `Blueprint Agent + Package Agent`
4. 复用现有工件输出区，在右侧新增“本次提取素材”
5. 复用现有资源库和模板库的存储思路，为素材台新增专门对象，而不是继续把素材塞进 memory
6. 保留现有 `Reviewer / Critic` 质量层，只在通过后追加 `Asset Extraction`

## 6. 教学法落地：UbD + PBL

### 6.1 方法论关系

本方案采用“UbD 做总骨架，PBL 做活动组织方式”的设计。

面向教师的产品展示语言建议统一为：

- `目标蓝图`：对应内部实现的 `UbD Stage 1`
- `证据设计`：对应内部实现的 `UbD Stage 2`
- `活动流程`：对应内部实现的 `UbD Stage 3`

- UbD 负责定义：
  - 预期学习结果
  - 可接受证据
  - 学习活动安排
- PBL 负责定义：
  - 驱动性问题
  - 真实任务情境
  - 项目里程碑
  - 学生产出与展示方式

### 6.2 最终输出结构

每次生成结果建议固定包含以下内容：

1. 课程概览
2. 目标蓝图（内部对应 `UbD Stage 1`）
3. 证据设计（内部对应 `UbD Stage 2`）
4. 活动流程（内部对应 `UbD Stage 3`）
5. PBL 项目设计
6. 教案初稿
7. PPT 大纲
8. 学具附录
9. 资料与延伸建议

### 6.3 学具附录结构

第一版采用“课程为主，学具为辅”的策略。学具不是独立产品主线，而是课程结果里的一个固定附录。

学具附录建议固定包含以下字段：

1. 学具名称与一句话用途
2. 服务的教学目标
3. 对应课堂环节
4. 材料清单
5. 准备方式
6. 制作或组装说明
7. 课堂使用方法
8. 安全与替代方案
9. 可选的定制或打样建议

第一版原则：

- 默认优先校内可落地材料
- 允许提出定制或打样建议
- 但不把复杂定制流程做成独立系统
- 每个学具方案都必须明确服务哪个教学目标和课堂环节

### 6.4 教学法检查规则

Lead Agent 在最终汇总时至少检查以下问题：

- 学习目标是否清晰
- 评价证据是否对应学习目标
- 学习活动是否支撑评价任务
- PBL 任务是否具有真实问题驱动
- AI 教育或科学教育内容是否落到真实情境

### 6.5 课程质量控制层

第一版不采用“单个黄金案例 + 固定模板”的质量控制方式，而采用：

- 少数不可妥协项
- reviewer / critic 的 rubric 检查
- 多案例校准

这一层的详细规则不再继续堆叠在本产品方案中，而是单独落到：

- `docs/standards/course-generation-quality-spec.md`
- `docs/cases/case-01-animal-vision.md`

这样做的原因是：

- 让产品方案继续负责系统结构，不承担过细的执行规则
- 让质量规范独立存在，避免 agent prompt 被模板化
- 让案例库承担“校准为什么好”的作用，而不是规定课程必须长什么样

当前实现状态（V1.3）：

- `Reviewer` 与 `Critic` 均已接入主流程（`Presentation -> Reviewer -> Critic -> 草案评审点`）
- 草案评审点支持 Reviewer/Critic 冲突摘要与更保守返工建议

## 7. 系统架构

### 7.1 总体结构

建议采用最小可演示架构：

- 前端：教师工作台
- DeerFlow 后端：多代理编排与记忆
- 外部工具层：白名单搜索、课程资料输入、少量文件输出

### 7.2 前端职责

前端只做必要展示，不承担复杂推理逻辑。建议保留四个核心区域：

- 任务输入区
- 代理执行过程区
- 结果展示区
- 教师记忆区

其中最重要的是“代理执行过程区”，因为这是学习 DeerFlow 的关键窗口。
对于 `Human in the loop`，前端不需要单独做复杂流程编排器，第一版只需要在执行过程中展示清晰的审批卡片和继续按钮。

### 7.3 后端职责

后端由 DeerFlow 承担核心编排职责：

- 接收教师任务
- 读取相关记忆
- 拆解子任务
- 调度多个子代理
- 汇总结果
- 写入高价值记忆

## 8. 多代理设计

### 8.1 设计原则

第一版不使用完全自由编排，而采用“半固定多代理编排”：

- 角色固定
- 分工清晰
- 输入输出标准化
- 便于观察与调试
- 运行层尽量少，概念层可以细
- 不把每个教学环节都做成一等 agent

### 8.2 代理角色

建议采用 `4+1` 运行模型：

1. `Lead Agent`
职责：
- 读取用户输入、记忆、素材与团队规则
- 生成任务简报
- 确认本轮生成策略
- 调度下游运行角色
- 发起审批与返工
- 在结尾推动素材入库与记忆写入

2. `Blueprint Agent`
职责：
- 合并目标蓝图与研究支撑
- 提炼大概念、核心问题、驱动方向与项目方向
- 输出老师可锁定的课程蓝图

3. `Package Agent`
职责：
- 合并证据设计、活动流程、学具附录与最终整理
- 直接生成完整课包
- 保持目标、证据、活动和学具之间的一致性

4. `Reviewer Agent`
职责：
- 基于硬门槛与 rubric 对课程包做首轮质量评审
- 输出结构化评审摘要供草案评审点使用

5. `Critic Agent`
职责：
- 作为可选的第二层质量复核
- 挑战 Reviewer 盲点，识别潜在漏判风险
- 在必要时给出更保守的返工建议

同时保留两类系统能力，但不把它们做成一等 agent：

- `Asset Retrieval`
  - 在生成前召回教师素材台中的高相关素材
  - 输出推荐吸收素材、冲突素材和优先复用建议
- `Asset Extraction`
  - 在草案评审通过后抽取可复用素材候选
  - 输出候选素材清单、推荐分类、来源位置与入库理由

概念层仍然可以细分为：

- `目标蓝图`（内部对应 `UbD Stage 1`）
- `证据设计`（内部对应 `UbD Stage 2`）
- `活动流程`（内部对应 `UbD Stage 3`）
- `学具附录`
- `最终整理`

但这些概念层在第一版不必都成为独立运行角色。

### 8.3 Lead Agent 角色

Lead Agent 负责：

- 读取用户输入
- 读取教师记忆
- 读取相关素材与参考资料
- 生成任务简报与生成策略
- 调度运行角色并汇总结果
- 做最终一致性检查
- 决定本轮优先生成还是优先复用
- 在通过后发起素材提取确认
- 决定哪些内容值得写入记忆

### 8.4 Agent 协作流程

这套系统里的 agent 配合方式，不是“多个 agent 同时讨论”，而是“Lead Agent 统筹分工，子代理按职责产出结果”。

第一版建议采用“部分并行 + 部分串行”的协作方式：

1. `Lead Agent` 读取任务、记忆和技能
2. `Asset Retrieval` 先召回相关素材与参考资料
3. `Lead Agent` 形成任务简报并确认生成策略
4. `Blueprint Agent` 生成课程蓝图
5. `Lead Agent` 发起蓝图确认
6. `Package Agent` 生成完整课包
7. `Reviewer Agent` 输出质量评审结论
8. 视风险与置信度决定是否启用 `Critic Agent`
9. `Lead Agent` 发起最终评审并决定是否局部返工
10. 通过后由 `Asset Extraction` 输出候选素材并进入入库确认

概念层依赖关系仍保留，但收在运行角色内部：

- `Blueprint Agent` 内部覆盖目标蓝图与研究支撑
- `Package Agent` 内部覆盖证据设计、活动流程、学具附录与最终整理
- `Asset Retrieval` 发生在生成前
- `Asset Extraction` 依赖草案评审通过后的完整课包

这样设计的原因是：

- 能保留 DeerFlow 的多代理学习价值
- 不会让依赖关系和运行成本同时膨胀
- 便于后续做“局部返工”，例如回退蓝图层或课包层

### 8.5 Human in the Loop 设计

第一版采用“关键节点审批式 HITL”，并且只允许 `Lead Agent` 发起人机审批。子代理不直接打断老师。

原因：

- 主代理最适合与教师对话和解释上下文
- 子代理更适合作为执行专员，避免频繁打断
- DeerFlow 现有的 `ask_clarification + ClarificationMiddleware` 机制天然适合由主代理发起审批

第一阶段建议保留 `3 个强审批点 + 1 个轻确认节点`：

1. `任务确认点`
在正式拆解前，由 `Lead Agent` 输出任务确认卡，内容包括：
- 课程主题
- 学段与课时
- UbD/PBL 方向
- 是否需要学具及学具限制条件
- 本轮生成策略是“从零生成”还是“优先吸收已有素材”

2. `课程蓝图锁定点`
在 `Blueprint Agent` 完成后，由 `Lead Agent` 输出目标与项目方向确认卡，内容包括：
- 大概念
- 核心问题
- 项目方向
- 是否需要调整研究重点

3. `草案评审点`
在 `Package + Reviewer + Critic` 完成后，由 `Lead Agent` 输出最终草案评审卡，内容包括：
- 课程草案摘要
- 学具附录摘要
- 可选择接受、局部重做或整体修改
- Reviewer/Critic 是否一致，以及冲突时的建议返工范围

4. `素材提取确认`
在草案通过后，由 `Lead Agent` 展示本次抽取出的候选素材，支持老师：
- 一键入库
- 跳过不合适素材
- 调整素材分类后入库

审批规则建议：

- 每次只审批一个节点
- 默认提供明确选项，而不是开放式追问
- 支持“局部返工”，优先重做单个 agent 的产出
- 对草案评审点启用返工护栏：第一次拒绝局部返工，第二次拒绝重开任务确认
- 只有最终接受后，再写入稳定偏好和可复用模板到长期记忆

第一版可以把 HITL 建立在 DeerFlow 现有的澄清中断机制上：

- `Lead Agent` 用 `ask_clarification` 发起审批
- 中间件中断执行
- 前端展示审批卡片
- 教师选择后继续执行后续 agent 流程

## 9. 记忆系统设计

### 9.1 记忆目标

这版记忆系统不追求“记住一切”，而追求“记住真正影响后续产出的内容”。

同时需要明确边界：

- `记忆` 负责保存偏好、连续性和团队风格
- `教师素材台` 负责保存可复用教学内容资产

也就是说，好的驱动问题、活动点子、学具方案和表达模板，优先进入素材台，而不是继续混在 memory 里。

### 9.2 三类记忆

1. `教师偏好记忆`

例如：

- 偏好探究式还是讲授式
- 更关注 AI 素养、科学探究还是工程设计
- 常用课时长度
- 偏好量规、项目展示或实验活动
- 偏好的学具形式和课堂组织方式
- 喜欢的输出风格

2. `课程连续性记忆`

例如：

- 已做过哪些单元
- 使用过哪些项目主题
- 已出现过的核心概念
- 哪些内容需要避免重复

3. `团队共享记忆`

例如：

- 小团队通用 UbD 模板
- 常见 PBL 项目结构
- 可复用的学具附录模板
- 统一术语和表达口径
- 校内推荐课程资源格式

### 9.3 记忆写入原则

只在以下情况下写入记忆：

- 教师明确表达稳定偏好
- 某次课程形成了可复用模板
- 某次学具附录形成了可复用方案
- 某种项目路径被反复采纳
- 管理员确认某条团队规则值得共享

### 9.4 记忆读取时机

在任务刚开始时读取：

- 教师个人偏好
- 最近课程上下文
- 团队共享模板

在最终汇总阶段再次参考：

- 是否需要沿用已有表达风格
- 是否应避免内容重复
- 是否要沿用既有课程连续性
- 是否沿用既有学具偏好和组织方式

## 10. 技能系统设计

建议把专业方法论沉淀为技能，而不是都写进一个超级 prompt。

第一版建议至少做以下技能：

1. `ubd-course-design`
- 说明 UbD 三阶段要求
- 输出结构模板
- 常见错误提醒

2. `pbl-learning-design`
- 说明驱动性问题设计方法
- 项目任务书结构
- 里程碑与评价建议

3. `ai-science-education`
- 面向人工智能教育和科学教育的课程语境
- 常见课程主题
- 适合教学的案例类型

4. `learning-kit-planning`
- 面向小学课堂的学具附录结构
- 低成本与教学适配性优先原则
- 材料清单、准备方式和使用建议模板

5. `course-quality-review`
- 说明不可妥协项
- 说明 reviewer / critic 的 rubric 用法
- 说明案例只能用于校准，不能用于复写

技能的价值不只是“让结果更好”，更重要的是让你学会如何把专业方法论注入 Agent 系统。

## 11. 前端功能建议

### 11.1 页面结构

建议做一个最小工作台界面：

- 左侧：任务入口、历史课包、教师素材台
- 中间：本次任务的执行流、审批卡与当前生成阶段
- 右侧：课程蓝图、教案、PPT、学具、参考资料与本次提取素材

### 11.2 必备功能

- 输入课程主题和约束条件
- 支持“快速生成课包 / 带着已有想法生成”两种入口
- 选择学段、课时数、课程方向
- 展示任务简报与生成策略
- 展示当前由哪个 agent 在执行
- 展示每个子代理的输出摘要
- 最终结果结构化呈现
- 展示本次读取到的教师记忆
- 展示本次召回的相关素材与本次提取素材
- 展示本次生成的学具附录
- 在关键节点弹出审批卡片

当前实现状态（V1.3）补充：

- 审批卡支持可选元字段：`checkpoint_id`、`recommended_option`、`retry_target`、`details`
- 聊天页右侧“教师记忆区”已落地，展示四类教育长期记忆与“本次使用信号（education_signals）”
- 草案评审已启用返工护栏：一次局部返工后再次拒绝将重开任务确认

### 11.3 可延后功能

- 拖拽式流程编排
- 复杂模板市场
- 权限管理
- 多组织空间
- 精细版资源库管理

## 12. 一次完整任务的数据流

1. 教师新建任务，并选择“快速生成课包”或“带着已有想法生成”
2. Lead Agent 读取教师记忆和团队共享记忆
3. `Asset Retrieval` 召回相关素材与参考资料
4. Lead Agent 生成 `任务简报`
5. Lead Agent 发起“任务确认 + 生成策略确认”审批
6. `Blueprint Agent` 生成 `课程蓝图`
7. Lead Agent 汇总生成 `课程蓝图`，并发起蓝图锁定审批
8. `Package Agent` 生成完整课包
9. `Reviewer` 完成首轮质量评审
10. 视风险与置信度决定是否启用 `Critic`
11. Lead Agent 发起草案评审点审批，并按质量规范聚焦硬门槛和冲突风险
12. 草案通过后，`Asset Extraction` 抽取本次可复用素材候选
13. 老师确认候选素材入库到“教师素材台”
14. 系统提取高价值偏好与连续性信息并写入记忆

这条链路要在前端尽量可见，便于你学习和演示。

## 13. 第一版成功标准

如果第一版达到以下标准，就算成功：

- 能稳定跑通一条完整课程生成流程
- 前端可展示多代理执行过程
- 至少有一类素材或记忆能明显影响下一次生成
- 产出内容体现 UbD 和 PBL 结构
- 有一套可复用的质量执行规范和案例校准方式
- 你能够解释每个代理做了什么、为什么这样分工

## 14. 风险与控制策略

### 14.1 风险

- 任务拆解过度自由，导致结果不稳定
- 记忆写得过多，出现脏数据
- 联网搜索内容质量不稳
- 前端做太重，挤占学习 Agent 的时间
- 一开始想覆盖太多教育场景，导致失焦
- 审批点过多，导致体验拖沓

### 14.2 控制策略

- 第一版采用固定代理角色
- 记忆写入从严控制
- 搜索与资料输入使用白名单来源
- 前端只做最小可演示工作台
- 专注一个主流程：UbD/PBL 备课生成
- 保持 3 个强审批点 + 1 个轻确认节点，不扩成全流程逐步审稿

## 15. 学习路径总览

你的主目标是学习 DeerFlow，而不是尽快做出商业化系统。因此学习路径应该围绕以下四个问题展开：

1. DeerFlow 如何定义和调度多代理
2. DeerFlow 如何读写长期记忆
3. 技能系统如何影响系统提示与工作流
4. 前端如何把 Agent 过程展示给人看

## 17. 推荐的实际学习顺序

如果你每次只学一个重点，建议按下面顺序来：

1. 跑通官方项目

3. 看懂 Subagent 调度机制

## 18. 你在学习中最该关注的几个问题

- Lead Agent 为什么决定这样拆任务
- 子代理之间哪些内容需要隔离，哪些需要共享
- 记忆究竟应不应该写入，写入什么
- 技能是改变结果质量，还是改变工作流
- 前端应该展示“答案”，还是展示“过程”

把这几个问题想明白，比多加几个功能更重要。

4. 画出这个 Demo 的信息架构和页面草图

## 20. 结论

这不是一个追求“大而全”的教育产品方案，而是一条适合学习 Agent 框架的落地路径。

选用 DeerFlow 的意义，在于你可以通过一个真实又收敛的教育应用场景，把多代理编排、记忆系统、技能体系和前端工作台一次性串起来。对当前阶段来说，这比做一个泛化的教育助手更有学习价值，也更容易做出真正像样的 Demo。
